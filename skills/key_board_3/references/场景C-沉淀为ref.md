# 场景 C：把特定主题沉淀为 reference（★ 核心场景）

> **这是 key_board_3 触发最频繁的场景**。当你听到用户说"把 xx 沉淀为 reference / 拆出去 / 单独存一下"，第一时间想到这个流程。

## 为什么这是核心场景

| 场景 | 触发频率 | 典型信号 |
|---|---|---|
| **C：沉淀特定主题到 ref** | ⭐⭐⭐⭐⭐ | "把错误案例库拆出去"、"把 format 规范沉淀一下"、"这部分单独存" |
| A：SKILL.md 膨胀拆分 | ⭐⭐ | "skill 太大了"（往往是 C 的子集） |
| B：加 ref 加载引导 | ⭐ | "加个 ref 引导"（常和 C 一起做） |
| D：多 skill 统一架构 | ⭐ | "统一一下"（低频，批处理型） |

## 与 key_board_2 的核心区别

| 维度 | key_board_2 | key_board_3（场景 C） |
|---|---|---|
| 动作 | **创建新 skill** | **创建 references/ 子文档** |
| 输出物 | 独立目录 `.claude/skills/<新skill>/` | 子目录 `<原skill>/references/<ref-name>.md` |
| 触发词 | "总结成 skill"、"做成技能" | "沉淀为 reference"、"拆到 ref"、"单独存" |
| 主 SKILL.md 改动 | 不存在 | 在原位置留锚点 + 一句话总结 + 末尾索引表登记 |
| 涉及目录 | 创建 1 个新目录 | 修改 1 个已有目录 |

> **一句话总结**：key_board_2 写新楼，key_board_3 拆老楼。两者互补不冲突。

## 场景 C 完整 SOP

### Step 1: 识别主题

读取目标 skill 的 SKILL.md，定位要沉淀的内容：

- 是哪一章？起止行号？
- 主题是否可以独立成立？（有明确触发时机 + 独立价值）
- 适合沉淀的常见主题：
  - 错误案例库（持续累积，独立价值高）
  - format 规范（写新 skill 时才查）
  - 调用细节 / API 速查（执行具体步骤时查）
  - 反思方法论（评估阶段才用）

**反例**：不要把"成功标准检查清单"沉淀为 ref——它每次创建 skill 都要用，跟着主流程走才对。

### Step 2: 设计 ref 文档结构

```markdown
---
name: <ref-name>
description: <一句话说清这个 ref 是干什么的、什么时候读>
---

# <Ref 标题>

## 加载时机
（什么时候必须读这个 ref？主 SKILL.md 哪个流程步骤？）

## 详细内容
（完整迁移过来的内容）

## 关联引用
（用 [[xxx]] 链接到主 SKILL.md 或其它 ref）
```

**注意 frontmatter 的 description**：写"什么时候读"，不要写"内容总结"。

### Step 3: 迁移内容

- 把原 SKILL.md 中的目标章节**完整**复制到 `references/<ref-name>.md`
- 复制时保持原有结构不变
- 修正内部链接：用 `[[ref-name]]` 形式（如果有引用其它 ref）
- 删除：任何已下沉的过时的、重复的、对其它 ref 的反向引用

### Step 4: 在主 SKILL.md 留锚点

在原章节位置（不要删除章节标题，只清空内容或留一句总结），写：

```markdown
详见 [[<ref-name>]]。
```

或者更详细：

```markdown
> 📎 **本节详细内容已沉淀到 [[<ref-name>]]**。
> 
> 何时读：[填写具体的加载时机，如"创建新 skill 时 / 评估阶段 / 用户明确要求时"]
```

### Step 5: 在主 SKILL.md 末尾登记索引

如果还没有 ref 索引表，新增一个章节：

```markdown
## 引用索引（按需加载）

| ref | 何时读取 | 路径 |
|---|---|---|
| [[<ref-name>]] | [一句话说明加载时机] | references/<ref-name>.md |
| [[<ref-name-2>]] | [一句话说明加载时机] | references/<ref-name-2>.md |
```

如果已有索引表，补一行即可。

## 场景 C 完成后必须验证

- [ ] `references/<ref-name>.md` 已创建
- [ ] 文件 frontmatter 完整（name + description）
- [ ] description 写"什么时候读"，不是"内容总结"
- [ ] 主 SKILL.md 原位置有 `[[ref-name]]` 锚点
- [ ] 主 SKILL.md 末尾 ref 索引表已登记
- [ ] 主 SKILL.md 行数有所下降（不一定要 < 500，但要明显改善）
- [ ] 核心流程/触发条件**没有**改动

## 场景 C 常见错误

| 错误 | 后果 | 预防 |
|---|---|---|
| 把"成功标准"也拆出去 | 创建流程失去完整性 | 只有低频、专项内容才拆 |
| ref 内不放加载时机 | AI 不知道什么时候读它 | 每个 ref 文档开头必有一段"加载时机" |
| 拆完忘记登记索引表 | ref 成为孤儿文档 | Step 5 强制登记 |
| 把主 SKILL.md 章节标题也删了 | 失去导航锚点 | 只清空内容，标题保留 |
| 拆完主 SKILL.md 行数反而更多 | 加了过多引导段 | 引导段用一句话，不展开 |

## 场景 C 的反例（不该拆的情况）

- ❌ 主流程步骤 1/2/3/4/5 → 主流程是骨架，拆散就废了
- ❌ 触发条件 + 核心原则 → 这两块是 skill 的"身份证"，必须保留在主 SKILL.md
- ❌ 成功标准检查清单 → 每次创建 skill 都要对照
- ❌ < 100 行的小章节 → 拆出去反而增加跳转成本

## 场景 C 的正例（该拆的情况）

- ✅ 错误案例库（高频累积、内容多、有独立价值）
- ✅ 格式规范（写 frontmatter 时才查、用时短）
- ✅ API 速查（执行具体步骤时按需查）
- ✅ 反思方法论（评估阶段才用、可以独立深挖）
- ✅ 子领域/子框架的差异化细节（多领域 skill 才需要）
