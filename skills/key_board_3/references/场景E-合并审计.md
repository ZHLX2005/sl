# 场景 E：合并候选审计（Exploration-Mode Audit）

> **使用条件**：用户提出"探索 / 观察 / 扫描"类诉求，但不指定合并目标。本场景是**只出报告、不动手**的探索模式。

## 为什么单独成 E 场景

| 场景 | 输入 | 输出 | 行为模式 |
|------|------|------|---------|
| A/B/C/D | 用户指定一个 skill 和动作 | 修改后的 skill | **定向改一个** |
| **E** | 用户要求"扫所有 skill" | **一份合并候选报告** | **横向扫 N 个** |

E 不直接执行 A/B/C/D。它是它们的**前置发现环节**——报告交给用户决策后，再按报告条目触发对应的 A/B/C/D 流程。

## 触发条件

- "扫一遍现有 skill"、"看看哪些 skill 可以合并"
- "skill 是不是有重复的"、"主题相似的 skill"
- "哪些 skill 可以合并"、"有可合并的轻量 skill 吗"
- "skill 之间有没有相互协作的关系"
- 用户明确说"探索模式"、"审计模式"、"scan skills"

**关键判断**：用户说"观察 / 探索 / 看看" 时默认进入 E 场景。用户说"合并 x 到 y" 时直接进合并场景（特殊优化）。

## 何时**不**触发本场景

- 用户已明确指定两个 skill 要合并 → 走合并场景（特殊优化）
- 用户要求分析单个 skill 的结构 → 走场景 A/B/C/D
- 用户要求创建新 skill → 走 key_board_2

## 审计的 5 个维度

对每个候选 skill 对，从以下 5 个维度独立打分（0-3）：

| 维度 | 含义 | 检查方法 |
|------|------|---------|
| **主题相似** | 两个 skill 的 frontmatter description 主题词重合度 | 提取核心名词（CSS、OCR、sidebar……），统计共现 |
| **场景相似** | 都在"新增/修改某物"时被触发 | 比 description 里的触发条件；同动词（修改/编写）同宾语（某模块）= 高分 |
| **相互协作** | 一个 skill 在 description 或正文里 [[]] 引用另一个 | 串读 SKILL.md 全文的 `[[xxx]]` 链 |
| **轻量合并价值** | 一个或两个都很小（< 100 行） | 行数 + 章节数；小文件合并不浪费跳转成本 |
| **结构倾向**（ref 信号） | 是否有 `references/` 子目录 | 见下方"结构倾向判断规则" |

### 结构倾向判断规则（关键信号）

**这是合并方向的最强信号**——比行数、主题词都更可信：

| A 状态 | B 状态 | A 分 | B 分 | 含义 |
|--------|--------|------|------|------|
| 无 references/ | 无 references/ | 0 | 0 | 双方都是"扁平静态"，没有特化指导积累 |
| **无 references/** | **有 references/** | **3** | **1** | **A 极适合并入 B**（A 是轻量独立文档，B 已是 ref 架构主人） |
| 有 references/ | 无 references/ | 1 | 3 | B 适合并入 A（对称） |
| 有 references/ | 有 references/ | 2 | 2 | 双方都成熟，**慎合并**（会破坏各自的 ref 架构） |

**核心原则**：

- ⚠️ **无 references/ 的 skill 极容易被合并** — 主 SKILL.md 是单一文档、没有特化分层的 skill，最容易被特化场景并入某个成熟 skill 的 ref 里
- ✅ **有 references/ 的 skill 适合做合并接收方** — 已经有"主文档 + 特化 ref"架构，意味着积累了多个特化场景，再吸收一个同主题轻量 skill 是自然演化
- 🚫 **双方都有 references/ 时合并要谨慎** — 两份成熟产物合并会互相破坏 ref 树的清晰性，应优先用"互引"替代"硬合并"

## 判断阈值

按 5 维总分定等级（注意：现在总分范围 0-15）：

| 总分 | 等级 | 建议动作 |
|------|------|---------|
| 0-4 | **弱相关** | 建议保留独立 |
| 5-8 | **可合并** | 提示用户；列出合并后导航可能的变化 |
| 9-11 | **强建议合并** | 主动给出方案（合并到哪个、ref 名、行数预估）|
| 12-15 | **必须合并**（如有 ≥2 维度高分）| 直接给三套方案让用户选 |

**额外硬约束**（任一触发即降级或否决）：
- 任意一方 ≥ 300 行 → 不建议合并
- 双方都已成熟（有 references/）→ 优先用 **互引**，拒绝硬合并（破坏各自的 ref 树）
- 一个已是"通用技术" 主题（适用 ≥ 3 个不同模块）→ 不并入单个模块 skill
- 结构倾向分双方都是 0 → 合并后接收方需要先拆分（增工作），降一级

## 报告模板

输出一份 Markdown 报告，章节如下：

```markdown
# 合并候选审计报告 — [项目名]

> 扫描对象：[N] 个 skill，扫描时间：[日期]，扫描模式：探索式

## 评分总览

| Skill A | Skill B | 主题 | 场景 | 协作 | 轻量 | 结构 | 总分 | 等级 |
|---------|---------|------|------|------|------|------|------|------|
| A (无 ref) | B (有 ref) | 2 | 1 | 2 | 1 | 3 | 9 | 强建议合并 |
| ... | ... | ... | ... | ... | ... | ... | ... | ... |

## 结构地图（必须有，先于详细分析）

扫描一开始先列出**每个 skill 的 ref 状态**，配合总览读：

| Skill | 行数 | 是否有 references/ | ref 数量 | 角色定位 |
|-------|------|-------------------|---------|---------|
| skill-A | 120 | ❌ 无 | 0 | 扁平静态 — 易被合并 |
| skill-B | 252 | ✅ 有 | 3 | 主文档 + 特化 ref — 适合作接收方 |
| skill-C | 454 | ✅ 有 | 2 | 成熟 skill — 慎合并 |
| ... | ... | ... | ... | ... |

> **结构地图是合并方向的指南针**：看到"无 ref"就先考虑它能不能并入哪个"有 ref"的同主题 skill。

## 每个候选对的详细分析

### 候选 1：[A] ↔ [B]（总分：9，强建议合并）

**主题相似（2/3）**
- A 的 description 提到："X、Y、Z"
- B 的 description 提到："X、W、Y"
- 共现：X、Y

**场景相似（1/3）**
- A 触发条件："修改某物 A"
- B 触发条件："构建某物 A"
- 重合动词："A"，但上下文略不同

**相互协作（2/3）**
- A 的 SKILL.md 有 `[[B]]` 引用 1 处
- B 的 description 提到 A 1 次

**轻量合并价值（1/3）**
- A 行数：120
- B 行数：80
- 都不够小，合并后约 200 行，可接受

**结构倾向（3/3）** ★ 关键信号
- A：❌ 无 references/ — 扁平静态
- B：✅ 有 references/ (3 个) — 成熟主文档
- **合并方向极强**：把 A 整文件沉淀为 B 的 `<ref-name>.md`

**建议方案**
- 把 A 沉淀为 B 的 references/<ref-name>.md
- B 的 SKILL.md 加锚点 + 索引行
- A 目录删除
- B 的 description 需补 A 的触发词

**潜在风险**
- A 现有用户习惯要切换路径
- 跳转深度从 0（独立 skill）→ 1（ref），但收益是 B 的 ref 树更完整

## 不动条目（弱相关或高风险）

| Skill A | Skill B | 总分 | 不动理由 |
|---------|---------|------|---------|
| A (有 ref) | B (有 ref) | 6 | 双方成熟，硬合并破坏 ref 树，改用互引 |
| ... | ... | ... | ... |

## 整体建议

（顶层摘要：哪些该合并、哪些保留、哪些改互引、整体方向）
```

## 报告完成后做什么

E 场景**不直接动手**。报告交给用户：

1. 用户看完报告
2. 用户选定要执行的合并条目
3. 每个条目按对应场景（A/B/C/D 或合并场景）的 SOP 执行
4. E 报告作为**审计记录**可归档到 `references/audit-history/` 或类似路径

## 反例（不该用 E 时）

- ❌ 用户已经指定"合并 A 到 B" → 直接合并，不出报告
- ❌ 项目只有 1-2 个 skill → 没横向扫描必要
- ❌ 想对 1 个 skill 做结构优化 → 走 A/B/C/D

## 正例（该用 E 时）

- ✅ 项目 ≥ 5 个 skill，主题词有重叠但分散
- ✅ 发现"轻量 skill 散落在多个角落" — 这种"无 ref"型 skill 是结构倾向维度的高分候选
- ✅ 看到某个成熟 skill（有 ref）周围围绕多个轻量同主题 skill — 明显应该归并
- ✅ 周期性清理（每个 release 或每季度一次）
- ✅ 新人接手 skill 库时的全局盘点

## 与 key_board_3 主流程的衔接

E 场景**不取代** A/B/C/D。它是它们的**前置发现层**：

```
E（审计报告）
    ↓ 重点关注结构地图里"无 ref"的 skill
    ↓ 用户选定条目
A/B/C/D 或合并场景（按条目执行）
    ↓ 完成后
E 报告归档（可选）
```
