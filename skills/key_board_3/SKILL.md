# Key Board 3 — 实验性功能探索沉淀（lab 探索 → 可重构 skill）

## 一句话定位

- **key_board_2**：通用"对话 → skill"元模板，沉淀的是**对话流程**。
- **key_board_3（本 skill）**：专做一件事——把一个**实验性功能**（通常是 `lab/` 下的探索模块）的**现状 + 搭建思路 + 关键代码索引 + 决策节点思路**，沉淀成一份新手能看懂、重构能照着走的 skill。

> 实验代码的生命周期是"试错"。试错中最值钱的不是最终代码，而是**走到哪了 / 为什么走这条路 / 哪条路被否决了 / 哪几行是灵魂**。本 skill 就是把这些"节点状态"固化下来，防止下次重构时把踩过的坑再踩一遍。

## 触发条件

当用户说以下内容时触发：

- "保留实验" / "保留这个实验" / "保留现在的探索状态"
- "沉淀实验" / "沉淀这个 lab" / "把实验存成 skill"
- "总结实验现状" / "总结一下这个实验"
- "保留探索节点" / "保留决策思路"
- "为重构保留参考" / "重构参考"
- "为新手整理这个实验" / "方便新手学习这个实验"

## 与 key_board / key_board_2 的边界（别用错）

| Skill | 沉淀对象 | 侧重点 | 什么时候用 |
|-------|---------|--------|-----------|
| key_board_2 | 任意对话流程 | 通用流程 → skill | 普通功能、对话复盘 |
| **key_board_3** | **实验性功能（lab/ 探索模块）** | **关键代码索引 + 决策节点思路** | **试错中的实验、需要重构参考的探索** |
| key_board_loop | 定期反思触发器 | 定时提醒 | 由 loop 工具驱动 |

判断标准：如果沉淀对象是 `lab/` 下的实验模块、或者是一个"还在试、可能推翻重来"的功能 → 用 **key_board_3**。

## 核心生成规则（与 key_board_2 完全一致，不变）

每次创建新 skill 时，必须：

1. 创建**独立目录** `.claude/skills/<新skill名称>/`
2. 在目录内创建 `SKILL.md`，必须包含 YAML frontmatter
3. **先调用 `/sc:reflect` 反思**，再动手
4. **Capture Intent**：先回答意图三问再写
5. 包含**错误案例**警示

> 上面这 5 条是"骨架"，key_board_2 已讲透，本 skill 不重复展开。本 skill 的增量在下面：**沉淀流程的 Step 5 / Step 6**——即"实验沉淀到底要写什么、怎么写"。

## 沉淀流程

### Step 1: 调用 /sc:reflect 反思（保留，但反思重点改为"实验视角"）

实验性探索的反思，重点回答：

- 这个实验想**验证什么假设**？验证到哪一步了？
- 当前能跑通的**最小路径**是什么？还卡在哪？
- 走过哪些**岔路**？哪条被否决了、为什么否决？（这是节点思路的原料）
- 哪几段代码是**"灵魂"**——换掉它整个实验就崩？

### Step 2: Capture Intent（保留三问 + 实验专属一问）

基础三问（解决什么问题 / 何时触发 / 输出格式）之外，**必须多回答一问**：

- **这个实验的"一句话现状"是什么？**（例："已实现 X25519+Ed25519 双密钥身份，握手未通"）

这一句话会直接写进生成 skill 的第一节，是新手最先看的东西。

### Step 3: 创建目录

```bash
mkdir -p .claude/skills/<实验沉淀skill名称>/
```

> 命名建议：`lab_<模块>_探索沉淀` 或 `<模块>_exploration`，让人一看就知道是实验沉淀。

### Step 4: 编写 SKILL.md（frontmatter 不变）

```yaml
---
name: <实验沉淀skill名称>
description: <触发描述：何时用、解决什么。不是内容总结>
---
```

### Step 5: 按实验沉淀模板写入（★ key_board_3 的灵魂）

生成的新 skill **必须包含以下 7 节**，顺序固定。这是 key_board_3 区别于 key_board_2 的核心——重心从"流程"挪到"代码索引 + 节点思路"。

#### ① 现状一句话（最先写）

一句话说清"现在到哪了"。新手第一眼就要看到这个。例如：

> 已用 Go 1.25 stdlib 实现 X25519（加密）+ Ed25519（签名）双密钥身份派生；握手协议未通，停留在单向认证。

#### ② 搭建思路（对标谁、为什么这么搭）

- 对标了哪个成熟方案？（例：对标 Happy 的双派生）
- 为什么选这条路？关键取舍是什么？
- 整体结构图（可用 Mermaid）。

#### ③ 关键代码索引（最重要 ★★★）

**这是本 skill 最重要的产出。** 把实验的"灵魂代码"逐条列成 `file:line` 索引表，让人不用通读全仓就能定位核心。格式见下方[关键代码索引规范](#关键代码索引规范最重要)。

#### ④ 决策节点思路（节点性质 ★★）

实验中每个"岔路口"都是一个**节点**。把每个节点记录成：**选了什么 / 否决了什么 / 为什么**。这是防止重构时重复踩坑的关键。格式见下方[决策节点思路规范](#决策节点思路规范节点性质)。

#### ⑤ 新手关键点（方便学习）

列出新手最容易卡、最容易理解错的 3–5 点。每点一句"坑 + 正解"。

#### ⑥ 重构参考点（用于重构）

如果将来要重构 / 正式化这个实验，应该注意什么？哪些是临时方案必须替换？哪些是可以保留的核心？

#### ⑦ 错误案例（保留 key_board_2 规范）

`| 错误操作 | 实际后果 | 正确做法 |` 表格，记录实验中真实踩过的坑。

### Step 6: 选择性创建 reference/ 参考文档（新增，可选）

当满足以下任一条件时，在 skill 目录下创建 `references/` 子目录，把详细内容拆出去（保持 SKILL.md 精简）：

- 对标方案的原理说明较长（如 E2E 握手协议细节）→ `references/protocol.md`
- 决策节点思路超过 10 个 → `references/decisions.md`
- 关键代码索引覆盖多个子模块 → `references/code_index.md`

SKILL.md 里只留索引和指针，详细内容指向 references。**这是渐进式披露**：主文件让人快速理解，细节按需读取。

## 关键代码索引规范（最重要）

每条索引必须能让读者**跳到具体行**看到灵魂代码。固定表格格式：

```markdown
## 关键代码索引

| 位置 | 是什么 | 为什么是灵魂 |
|------|--------|-------------|
| `lab/auth_cli/identity.go:14-21` | `Identity` 结构体：X25519+Ed25519 双密钥 | 整个实验的数据模型核心，换掉就崩 |
| `lab/auth_cli/identity.go:33` | `NewIdentity` 双派生入口 | 对标 Happy 的 deriveContentKeyPair |
| `lab/dag/graph.go` | DAG 图结构 | 调度模型的骨架 |
```

要点：

- **位置列必须可点击**：用反引号包 `path:line` 或 `path`。
- **"为什么是灵魂"列是关键**：不是抄注释，而是说清"这段代码为什么不能随便动"。
- 只索引"灵魂"，不要把整个模块每行都列上来——索引的价值在于**浓缩**。

## 决策节点思路规范（节点性质）

每个决策节点用一个**决策树片段**或**三栏表**记录。重点是"否决了什么、为什么"，而不只是"选了什么"。

格式 A——决策树（适合有分支的探索）：

```markdown
## 决策节点

### 节点 1：身份密钥如何派生？
- ✅ 选：一个 32B 种子同时派生 X25519 + Ed25519（对标 Happy）
- ❌ 否决：两套独立随机密钥 —— 否决原因：跨设备配对时分发两份密钥太复杂
- ❌ 否决：只用 Ed25519 —— 否决原因：签名密钥不能直接做 ECDH 协商
```

格式 B——三栏表（适合节点多、需要紧凑）：

```markdown
| 节点 | 选择 | 否决项及原因 |
|------|------|-------------|
| 密钥派生 | 单种子双派生 | 独立双密钥（分发复杂）/ 仅 Ed25519（无法 ECDH）|
```

> 为什么强调"否决项"：实验最怕"重构时把当年否决过的方案又当成新点子重新试一遍"。把否决理由留下，等于给未来的自己省一轮试错。

## 易错和坑（实验沉淀专属）

在 key_board_2 通用坑之外，本 skill 高频踩的坑：

| 错误 | 根因 | 预防 |
|------|------|------|
| 关键代码索引只写文件名不写行号 | 偷懒 | 必须写到 `path:line`，否则重构时找不到 |
| 索引把整个模块每行都列上 | 没理解"灵魂"概念 | 只列"换掉就崩"的代码，索引价值在浓缩 |
| 决策节点只写"选了什么" | 忘了否决项才是防重复踩坑的关键 | 每个节点必须有"否决项及原因" |
| 现状一句话写成功能介绍 | 混淆"现状"和"功能" | 现状 = 现在到哪一步了（含未完成部分）|
| 把 key_board_3 当 key_board_2 用 | 没区分沉淀对象 | 实验模块用 key_board_3，普通功能用 key_board_2 |
| reference/ 塞进 SKILL.md 导致主文件臃肿 | 不懂渐进式披露 | 长内容拆到 references/，主文件只留指针 |
