# 未解析双链补全 · 内容规范

> 本规范适用于 obsidian-repair-unresolved-links skill 中步骤6（读上下文 → 生成内容）的内容生成。

---

## 一、根性原则

### 1.1 拒绝模板化结构

严禁以下结构：
```
- 简介
- 发展历程 / 背景
- 核心概念 / 功能
- 应用场景
- 总结与展望
```

这种结构本质上是**信息目录**，不是**知识传授**。读完之后读者只知道了"有什么"，不知道"为什么"。

### 1.2 追求本质追问

每一篇文章要回答的不是"这是什么"，而是：

- **它解决的是什么矛盾？**（存在的原因）
- **如果没有它，世界会怎样？**（存在的必要性）
- **它和相似概念的边界在哪？**（存在的独特性）
- **它内部有什么张力、有什么 trade-off？**（存在的复杂性）

### 1.3 具体 > 抽象，案例 > 定义

不要以定义开头。如果必须引入一个抽象概念，先给一个**具体的、可感知的锚点**。

```
❌ SQL 是一种用于管理关系型数据库的编程语言。
✅ 当你有一个 Excel 表格，里面有 100 万行客户数据，
   你想找出"上个月买了但没付款的上海用户"——
   Excel 卡死了，SQL 却能在 0.3 秒返回结果。这就是 SQL。
```

## 二、内容定位：四种写作模式

按照链接在源文件中的上下文，选择一种模式切入。**不要混合使用**。

### 模式 A：概念深潜（Theoretical Deep Dive）

**适用场景：** 链接指向一个理论概念（如"事务隔离级别"、"B+树"、"CAP定理"）

**结构：**

1. **学科根基** — 这个概念属于哪个子领域？这个问题为什么会被提出来？
   - 事务隔离级别 → 并发控制理论 → 多个事务同时读写同一数据，结果应该是什么？
   - B+树 → 磁盘I/O模型 → 一次磁盘寻道 10ms，如何让查找次数降至 3-4 次？
2. **思想实验** — 不借助公式，用自然语言推导出这个概念产生的必然性
3. **内部机制** — 原理层面的拆解（而非功能层面的列举）
4. **边界与灰度** — 这个概念在什么条件下不成立？有什么争议或衍生变体？
5. **第一性原理回退** — 如果从零开始设计，我为什么会得出同样的结论？

**反例（禁止）：**
```
- 事务隔离级别的定义
- 四种隔离级别（读未提交、读已提交、可重复读、串行化）
- 每种隔离级别可能的问题（脏读、不可重复读、幻读）
- 总结：根据业务需求选择
```

这只是一个**产品说明书**，不是知识。

**正例示意：**
```
"两个事务并发修改同一条记录"这个问题，
在单机时代不存在（单线程），
在分布式时代又有了新解法（乐观锁、MVCC）。
而事务隔离级别这个概念的迷人之处在于，
它用 ANSI 标准把"并发正确性"这个连续谱
强行切成了四个离散档位——但真实数据库
（比如 MySQL 的 REPEATABLE READ）实际的
行为与标准定义有微妙偏移。
这些偏移不是 bug，而是工程妥协。
...
```

### 模式 B：实践拆解（Practice Deep Dive）

**适用场景：** 链接指向一个工具、配置、命令、排错场景（如 "怎么配置连接池"、"OOM 排查"）

**结构：**

1. **真实场景锚点** — 一个具体的、发生过的问题或决策
   - "某天线上服务 OOM 了，你拿到 heap dump 发现 byte[] 占了 60%"
2. **决策树** — 不是"步骤 1 2 3"，而是**在什么情况下走哪条路**
   ```
   发现 OOM
   ├─ 是堆内存不足？
   │  ├─ 堆内对象太多 → 查 dominator tree
   │  └─ 堆外内存泄漏 → 看 pmap + gdb
   └─ 是线程栈溢出？
      └─ 检查线程数 + -Xss
   ```
3. **配置/参数的实际含义** — 不是翻译文档，而是这个参数背后是什么限制、多大代价
4. **典型翻车案例** — 你踩过的坑、在 PR review 中见过的错误用法
5. **设计哲学** — 这个工具的设计者在哪个点上做了关键取舍？

**反例（禁止）：**
```
- 什么是连接池
- 连接池的好处
- 常用参数说明（maxActive、maxIdle、minIdle）
- 最佳实践
```

### 模式 C：还原论拆解（Reductionist Anatomy）

**适用场景：** 链接指向一个内部机制、源码片段、算法过程

**结构：**

1. **从外部观察到的行为开始** — 用户看到的表象
2. **逐层"打开"** — 像外科手术一样，一层层剥开
   - 第 1 层：接口 / API 签名
   - 第 2 层：调用链 / 状态机
   - 第 3 层：数据结构
   - 第 4 层：内存布局 / 磁盘结构（如果适用）
3. **回到外部** — 每层内部机制如何解释最初观察到的行为？
4. **如果改了会怎样？** — 对机制的深入理解可以通过"改一行代码试试"来验证

### 模式 D：概念拓扑（Concept Topology）

**适用场景：** 链接指向多个概念之间的关系、对比、演进脉络

**结构：**

1. **问题领域定位** — 这些概念共同在解决什么问题？
2. **关系图谱** — 不是"A 和 B 的区别"表格，而是它们在解决方案空间中的位置
   ```
   CAP 定理的视角下：
     一致性保证 ← 事务隔离级别、分布式共识、乐观锁
     可用性保证 ← 读写分离、缓存、降级
     分区容忍性 ← gossip 协议、最终一致性
   ```
3. **演进逻辑** — 从简单方案到复杂方案，每一步是因为什么新的约束条件
4. **实践中的选择权** — 在什么场景下应该牺牲哪个维度？

## 三、防重复扫描规范

> 这是最重要的约束——生成内容前，必须先确认 vault 里是否已经存在相关内容。

### 3.1 同一主题目录下的子文章必须先读

当源 root.md 处于一个主题目录下（如 `内容/数据库/undolog.md`），或是某个 topic 文件夹中的 root 文件时：

**生成新笔记之前，先列出并阅读同目录下所有已存在的 `.md` 文件。**

```
内容/数据库/
├── root.md          ← 源文件，包含 [[Undo Log]] 等链接
├── Undo Log.md      ← 已存在！必须先读，防止重复
├── Redo Log.md      ← 已存在！必须先读，防止重复
├── 事务隔离级别.md   ← 已存在！必须先读，防止重复
└── Crash Recovery.md ← 已存在！必须先读，防止重复
```

**必须做的事：**
1. 列出同目录下的所有 `.md` 文件（排除 root.md 自身）
2. 逐篇阅读其标题、核心观点和关键段落
3. 确保新生成的内容**不与任何已有文件重复**

**允许的情况：**
- 已有文章讲了 "undo 是什么"，新笔记可以从 **"undo 做错了会怎样"** 切入
- 已有文章讲了 "redo 的配置参数"，新笔记可以讲 **"redo 的磁盘布局和源码实现"**
- 已有文章讲了 "隔离级别的定义"，新笔记可以讲 **"MySQL 实现与 ANSI 标准的偏移"**

**禁止的情况：**
- 已有文章介绍了 Undo 的基本概念，新笔记又从头介绍一遍 Undo
- 已有文章列出了四种隔离级别，新笔记又列一遍

### 3.2 vault 层面的查重

对于要生成的链接主题，先在 vault 中全局搜索确认：

```bash
grep -ri "主题关键词" 内容/
```

如果 vault 中其他目录已有相关笔记：
- 用 `[[双链]]` 引用它们，不要重新写相同内容
- 明确标注"参见 [[已有笔记]]"

### 3.3 重复的红线级别

| 重复程度 | 判定 | 处理 |
|---------|------|------|
| 核心论点完全一致 | ❌ 严重违规 | 必须换角度或放弃生成 |
| 部分关键论据重叠 | ⚠️ 需要修改 | 删除重叠段落，补充新的论证 |
| 举例/场景相似但结论不同 | ✅ 允许 | 在开篇说明与 [[已有笔记]] 的关系 |
|  用完全不同的视角讲同一个概念 | ✅ 鼓励 | 必须引用对方并说明视角差异 |

### 3.4 如果确实无法避免部分重叠

在笔记开头明确标注：

```
> **关联笔记：** [[Undo Log]] 从实现层面介绍了 InnoDB Undo 的结构。
> 本文重点不在"Undo 是什么"，而是"Undo 为什么会设计成这样"。
> 两篇配合阅读可以获得完整理解。
```

这样读者（和未来的 AI）都能理解两篇笔记的关系，不会视为重复。

## 四、内容质量检查清单

每写完一篇笔记，逐条检查：

### 深度检查

- [ ] 这篇笔记的核心结论是否可以用 1-2 句话讲清楚？如果不能，说明没想透彻
- [ ] 是否至少有一处"第一次知道"的信息？如果没有，内容太泛
- [ ] 是否问了一个"为什么"并真的回答了它？
- [ ] 是否有具体的数字、时间、规模、量级？（"很快" → "3ms vs 300ms"）
- [ ] 是否有 trade-off 的讨论？（没有银弹是最好的银弹）

### 独特性检查

- [ ] 这篇笔记放到网上、翻到别处，有没有不可替代的价值？
- [ ] 是否只是把官方文档翻译/复述了一遍？
- [ ] 有没有个人实践、踩坑记录、或独立思考的痕迹？

### 关联性检查

- [ ] 是否引用了 vault 中其他相关的笔记？用 [[双链]] 体现
- [ ] 是否指出了与其他概念的边界和交叉？

### 结构检查

- [ ] 是否采用了本规范四种模式之一？
- [ ] 是否出现了"简介-表格-展望"之类被禁止的结构？

## 四、写作技法

### 4.1 锚点前置法

第一段务必给读者一个**可以抓住的东西**。可以是：

- **一个具体问题**："假设你要把 1TB 的数据从 MySQL 迁移到 TiDB……"
- **一个反直觉现象**："MySQL 的 REPEATABLE READ 级别实际上不会产生幻读——和 SQL 标准说的不一样。"
- **一个历史转折点**："Google 发表 Spanner 论文之前，全局一致性的代价是……"

### 4.2 对比与张力

利用对比制造认知张力：

| 张力的两端 | 效果 |
|-----------|------|
| 直觉预期 vs 实际行为 | 打破假设 |
| 理论最优 vs 工程妥协 | 揭示 trade-off |
| 简单场景 vs 复杂边界 | 展示灰度 |
| 前辈方案 vs 现代方案 | 展现演进 |

### 4.3 量级感知

所有时间、空间、规模必须有具体的数量和对比：

```
❌ 慢 → 快
✅ 10ms（磁盘寻道）→ 0.1ms（内存访问）→ 100 倍差距
❌ 可以缓存大量数据
✅ 64GB 内存可以缓存约 2.1 亿个 300 字节的 key-value 对
```

### 4.4 代码/命令的"为什么"注释

如果引用了代码或命令，不要只写"执行什么"，还要写"为什么这样做"：

```
# 不好：只写功能
sudo systemctl stop mysqld

# 好：解释为什么
# 先 stop 而不是 kill -9，让 mysqld 有机会做 clean shutdown，
# 做一次完整的 checkpoint，下次启动时 crash recovery 时间最短
sudo systemctl stop mysqld
```

## 五、示例：同一主题的两种写法对比

**主题：Undo Log**

### ❌ 禁止的写法（简介-分类-功能-总结）

```
# Undo Log

## 什么是 Undo Log
Undo Log 是 InnoDB 中用于记录修改前数据的日志。

## Undo Log 的功能
1. 事务回滚
2. MVCC

## Undo Log 的类型
- Insert Undo Log
- Update Undo Log

## 总结
Undo Log 在 InnoDB 中扮演重要角色...
```

这是目录，不是笔记。

### ✅ 推荐的写法（模式 A + 还原论）

```
# Undo Log —— 为什么我们要记录"改之前长什么样"

## 矛盾的起源

事务的原子性要求：要么全做，要么全不做。
但性能要求：数据要尽早落盘，不能等事务提交。

这两个要求放在一起，就产生了一个根本矛盾——

"如果数据在事务提交前就写了进去，
回滚时怎么把它变回原来的样子？"

解决方案有两个：
1. No-Steal：事务提交前不允许脏页落盘 → 性能灾难
2. Undo Log：允许脏页落盘，但记录修改前的镜像 → InnoDB 的选择

## Undo Record 的内容设计——解决了什么问题

要支持回滚，Undo Record 至少要记录：
- 修改的是哪张表？（Table ID）
- 修改的是哪一行？（主键 Key Fields）
- 改之前的值是什么？（Update Fields / 历史值的 delta）

但 InnoDB 不止于此——它还在 Undo Record 里记录了 trx_id 和 rollptr。
为什么？

因为 MVCC 复用了 Undo。

## 复用：一个数据结构承担两个职责

trx_id → 这个版本是谁产生的，判断是否可见
rollptr → 沿着它可以找到更老的版本，构建 ReadView 需要的完整历史链

这解释了为什么 Undo Record 比"仅仅支持回滚"所需的信息多得多。
每多一个字段，都是为了另一个职责。
```

## 六、特殊场景

### 6.1 链接跨文件引用

如果同一个 [[链接名]] 出现在多个源文件中，优先选择**上下文最丰富**的源文件作为内容生成的依据，并可以额外引用其他源文件中的相关信息。

### 6.2 链接指向一个非常窄的概念

一个 10 个字就能说明白的概念（如"连接超时"）——不要硬凑 2000 字。
这种情况 3-5 段点透即可，把深度压在里面而不是铺开。

### 6.3 链接在源文件中已有详述

如果源文件本身就对这个概念解释得很充分了，新笔记应该：
- 换一个角度切入（比如源文件是工程视角，新笔记就从学术视角）
- 或者深入一层（源文件讲了"是什么"，新笔记就讲"源码怎么实现的"）
- **不要重复源文件的内容**
